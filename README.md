# 🇮🇳 AI Brain for India Lawyers

> **Free practice OS for every Indian solo advocate. Terminal-native. Local-first by default (Ollama + Qwen3 — nothing leaves your laptop). Cloud-LLM optional with the [Pseudonymisation Gateway](https://github.com/Wolfgangrush/pseudonymisation-gateway) sanitising PII before any prompt leaves the machine. Built by an Indian advocate, for Indian advocates.**

**Version:** 0.1.3 · **License:** MIT · **Publisher:** [Wolfgang_rush](https://github.com/Wolfgangrush) · **Engine:** Built on a local memory architecture

> ⚠️ **AI can make mistakes. Always verify the output.**
>
> This software generates assistive drafts and suggestions only. Every legal claim, citation, statute reference, procedural step, deadline calculation, and ground of relief must be independently verified by a qualified human practitioner before filing, advising a client, or relying on the output. The publisher accepts no liability for outputs used without verification.

> 🛡️ **Privacy primitive: PII pseudonymisation** via [pseudonymisation-gateway](https://github.com/Wolfgangrush/pseudonymisation-gateway) (Wolfgang Rush · MIT). This firm uses the `india` jurisdiction module + Indian-diaspora overlay for cross-jurisdiction PII coverage. Open-source · zero runtime deps · session-scoped · in-memory only · never writes PII to disk.


> **🧠 AI Brain that LEARNS.** Every session makes the next one smarter. Two built-in Claude Code skills power this: `/retrospective` saves what the firm learned at session end — every jurisdiction, statute, argument pattern, and procedural rule you touched is logged so the firm's knowledge compounds. `/wake` loads that accumulated context the next time you start, so you never begin from zero. The firm is your second brain, and it gets sharper with every case.

## 🛡️ Privacy posture (the honest version)

**Architecture — three pieces decide your privacy posture:**

**(1) Local-only state.** Your matters, clients, drafts, notes, audit logs, calendar entries, and configuration live on your laptop in `~/.ailawfirm_india/`. Local SQLite + ChromaDB. The publisher operates zero infrastructure and cannot access this folder. MIT licensed source you can audit (`grep -ri "telemetry\|analytics\|requests.post\|urlopen" ailawfirm_india/`).

**(2) LLM backend — you choose.** The default `connect-local` command configures Ollama + Qwen3 to run the language model on your laptop (truly nothing leaves). If you opt into a cloud-LLM tier (DeepSeek / Claude / Gemini) for quality reasons, see the tier table above for cost + privacy trade-offs.

**(3) Pseudonymisation Gateway — always-on for cloud mode.** When you configure a cloud-LLM provider in `~/.ailawfirm_india/config.json`, the internalised `PseudonymisationGateway` (source: `ailawfirm_india/pseudonymisation.py`) automatically substitutes real names, government IDs (Aadhaar · PAN · GSTIN · IFSC · ration card · driving licence · voter ID), contact identifiers (phone · email), and case references (Supreme Court · High Court · District Court · NCLT · DRT formats) with deterministic placeholders BEFORE the prompt leaves your machine. The placeholder ↔ original map lives in memory only (never written to disk; destroyed when the gateway goes out of scope). Cloud vendors see only the abstract structure of the matter; the user sees real values restored in the response.

**For actual client matters under attorney-client privilege** (Bharatiya Sakshya Adhiniyam 2023 Section 132 / Indian Evidence Act 1872 Section 126) → **prefer the Local Ollama tier**. If cloud mode is unavoidable for quality reasons, the Gateway sanitisation is your structural privacy primitive — but execute your client engagement letter's AI-disclosure clause first, and document the basis in your audit log.

**DPDP Act 2023:**
- **Section 8(5)** (reasonable security safeguards) is supported by Gateway sanitisation in cloud mode (structural pseudonymisation = meaningful technical safeguard).
- **Section 5-7** (notice + consent + lawful basis) remain YOUR obligations as Data Fiduciary — Gateway sanitisation does NOT discharge them.
- **Section 8(5) + Section 17(2)(a)** (legal-proceeding exemption) substantially covers most everyday advocate processing; the Compliance Officer agent flags edge cases.

**BCI Rule 36:** the Compliance Officer specialist checks your published language for solicitation/advertising risk before you publish — independent of the dual-mode privacy architecture; YOU remain responsible for compliance with all bar conduct rules.

The wedge: every other cloud-AI legal tool sends raw client PII to the LLM by default. Wolfgang Rush AI Brain — India ships Ollama-first AND ships the Gateway as the privacy primitive that closes the gap when you choose cloud mode for quality reasons.

## 🔄 How to update your firm

When a new version of AI Brain — India is published, you pull it in with **one command**. Your matter data + your project-root `CLAUDE.md` are **never touched** — only the firm's installed code, skills, and prompts refresh.

### Path 1 — Plain terminal

```
ailawfirm-india update
```

Under the hood this runs `pip install --upgrade git+https://github.com/Wolfgangrush/ai-law-firm-india.git`. After it finishes, restart any open `ailawfirm-india` session so the new skills + prompts load.

### Path 2 — Inside Claude Code

Type:

```
/update
```

Claude runs the same command for you and reports the result.

### Path 3 — Inside Gemini CLI

Type:

```
/update
```

Same outcome — Gemini calls `ailawfirm-india update` for you.

### When to update

- **The publisher tells you** a new version is out → update.
- **Monthly hygiene** → update once a month so you stay current on skills + bug fixes.
- **After hitting a bug** → first thing to try is updating, in case it is already fixed upstream.

### What does NOT update (by design)

- Your matter folders (`~/Desktop/<your-firm>/<matter>/...`)
- Your project-root `CLAUDE.md` (your customisations always win)
- Your `~/.ailawfirm-india/` config + palace data
- Your chosen AI model setup (Ollama · DeepSeek · Claude · Gemini)

Only the firm's installed Python code, skills, and template files refresh. Your practice is unaffected.

### One catch — existing users + new template rules

If a new version updates the template `CLAUDE.md` (the firm's standing rules), your project-root `CLAUDE.md` is preserved because your customisations always win. To see what changed in the template after an update:

```
diff CLAUDE.md "$(python3 -c 'import ailawfirm_india, os; print(os.path.join(os.path.dirname(ailawfirm_india.__file__), "templates/CLAUDE.md"))')"
```

Review the diff and merge what you want into your own `CLAUDE.md`.

## 🔋 v0.1.2 — Thermal hygiene for low-RAM Macs

If you're on a fanless 8GB Mac (M1/M2 Air, base Mac mini) and notice the laptop getting hot, throttling, or shutting down during long drafting sessions, v0.1.2 adds three opt-in tools:

| Command | What it does |
|---|---|
| `ailawfirm-india lite on` | Enables lite mode — defers per-draft `.docx` generation (pandoc is the biggest single CPU consumer per drafting turn) and skips MCP autostart. Drafts saved as `.md` during the session. |
| `ailawfirm-india finalize <matter>` | Batch-generates `.docx` for every `.md` file in a matter folder. Runs `nice`'d pandoc once per file. Skips files where `.docx` is already up-to-date. Use at end of session. |
| `ailawfirm-india doctor` | Reports charger state, lite mode, pandoc presence, Python version, and memory pressure. Use when the laptop feels hot or slow. |

Lite mode is **opt-in**. Default behaviour is unchanged — upgrading via `ailawfirm-india update` does not alter how the firm works unless you explicitly enable lite mode.

Quality is **unchanged**. `pandoc` is deterministic for the same input — the `.docx` produced by `finalize` at end of session is bit-identical to the `.docx` that would have been produced per-draft.

Cross-LLM parity: Gemini users get the same behaviour via `/lite`, `/doctor`, `/finalize` slash-commands.

Full walkthrough: [`docs/RUNNING_ON_LOW_RAM_MAC.md`](docs/RUNNING_ON_LOW_RAM_MAC.md).

## ⚖️ Compliance posture — Supreme Court e-Committee AI framework

This tool is **assistive legal-work infrastructure**, not autonomous decision-making software. Its operational posture is aligned with the Supreme Court of India e-Committee's stated position on AI in legal work.

> *"AI and digital tools must be used as supportive instruments and should not be allowed to override judicial reasoning."*
>
> — **Justice Rajesh Bindal**, Judge, Supreme Court of India
> [*Judicial Process Re-engineering and Digital Transformation*](https://www.sci.gov.in/press-release-dated-april-12-2026/) conference, 11–12 April 2026
> Organised by the Supreme Court e-Committee in collaboration with the Department of Justice, Government of India.
> ([Coverage — Law Trend](https://lawtrend.in/ai-must-not-replace-judicial-reasoning-warns-supreme-court-justice-rajesh-bindal/))

The same posture underpins the Supreme Court's own AI infrastructure for the judiciary:

- **[SUPACE](https://www.drishtiias.com/daily-news-analysis/ai-portal-supace)** — *Supreme Court Portal for Assistance in Court Efficiency.* AI-enabled assistive tool launched on 6 April 2021 by then-CJI S.A. Bobde. Provides legal research, fact extraction, document review, and drafting assistance to judges and legal researchers. **By design, SUPACE is not a decision-making system** — it processes facts and surfaces them to the human user. The Supreme Court has recommended adoption across all Indian High Courts.

- **[SUVAS](https://www.drishtijudiciary.com/current-affairs/supreme-court-vidhik-anuvaad-software-suvas)** — *Supreme Court Vidhik Anuvaad Software.* AI-powered translation tool launched in November 2019 by then-CJI S.A. Bobde. Translates judicial documents, orders, and judgments between English and ten Indian regional languages.

### What this tool does — and does not — do under that framework

**Does:**

- Run an 8-intent brain classifier + 7 specialist assistive agents (citation validation, court lookup, classifier, etc.) under user direction.
- Surface authoritative citations (AIR / SCC / SCC OnLine), court hierarchy data, and procedural anchors for human review.
- Operate as a research and reference assistant for solo-advocate workflows.

**Does NOT:**

- Generate final pleadings, advice, or filings autonomously.
- Substitute for advocate professional judgment.
- Replace human verification.
- Operate without an enrolled advocate retaining full professional responsibility for any work product.

**Every output from this tool must be advocate-owned and human-verified before any client use or filing.** The enrolled advocate using this tool retains full professional responsibility under the Advocates Act 1961 and the Bar Council of India Rules, including verification of facts, accuracy of citations, correctness of legal grounds, and signature on every pleading filed.

This is the same standard the Supreme Court itself applies to its own AI infrastructure (SUPACE / SUVAS): **AI as supportive instrument, never as decision-maker.**
